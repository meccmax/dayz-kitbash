How export works

        

          Export Mod asks for an output folder, then writes:

          • <ModName>/config.cpp

          • <ModName>/data/<category>/<class>.png for each textured item

          • build.ps1 next to it

          Then open PowerShell there and run ./build.ps1 to produce the PBO.
          If your browser blocks folder writing, you'll get a .zip instead.

How Kitbash works

      
      
    

    

      
What Kitbash is

      
Kitbash builds the config and textures for a DayZ clothing/gear mod — it generates
      config.cpp, types.xml, and a build script that packs a ready-to-deploy PBO.
      It is a config + texture tool. It does not create 3D models.


      
The basic flow

      
    
        
  • Pick a base item (left dropdown) — any vanilla item, or ＋ Custom / modded base… to inherit from another mod.
    • 
        
  • Set stats — cargo, thermals, armor, durability, weight, size.
    • 
        
  • Add a texture — drop/paste an image, or click 🖌 Paint / edit texture to paint it in-app over the vanilla UV layout.
    • 
        
  • Export Mod — writes <ModName>/config.cpp, your PNGs, and build.ps1.
    • 
        
  • Run build.ps1 in that folder — converts PNG→PAA and packs the PBO (needs DayZ Tools).
    • 
      


      
UV templates

      
To paint over the real vanilla layouts, run build-templates.ps1 once. It extracts the
      vanilla textures from your DayZ install into templates/ (needs DayZ + DayZ Tools). They are
      not shipped with Kitbash — they're Bohemia's game assets.


      
Loot / types.xml

      
Custom items won't spawn until they're in the economy. The types.xml tab generates the loot
      entries — merge them into your mission (tune the defaults in the Project tab).


      
Proxies & attachments — read this

      
A "proxy" (a holster, sheath, NVG mount, grenade slot…) has two layers:

      
    
        
  • Config layer — the attachments[] slots an item exposes.
        Kitbash does this (Advanced → Attachments / proxy slots), and its importer reads them.
    • 
        
  • Model layer — the actual proxy point baked into the item's .p3d,
        authored in Object Builder. Kitbash can't create this.
    • 
      

      
So: if a base model already has the proxy points (a vanilla vest's grenade slots, a
      modded item you inherit from), Kitbash wires the slots and you're done. Adding a brand-new proxy
      point to a model needs that model in Object Builder — Kitbash only writes the config around it.


      
Modded / custom-model clothing

      
If your mod ships its own model (its own .p3d), use Advanced → Full modded-clothing config:
      set the model path, ClothingTypes, inventory slot, proxy/attachment slots, armor, repair, etc.
      Kitbash generates the entire config around a model you supply. You bring the .p3d; Kitbash does the rest.


      
Import an existing mod

      
Import .cpp reads an existing config.cpp back into editable items — base, stats,
      proxy slots, damage tiers, modded-clothing fields. Note: textures and .rvmat files aren't in
      the config, so re-attach them before exporting.


      
Good to know

      
    
        
  • Work is autosaved in this browser; use Save .json to back up or share a project.
    • 
        
  • Fully offline — no network, no telemetry. See the repo's SECURITY.md.
    • 
        
  • Run it from localhost (e.g. run.bat) so templates load and folder-export works.
